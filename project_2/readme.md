## Chapter I — Рекомендации к проекту

Этот документ — твоя главная точка опоры на протяжении всего проекта. Никаких лекций, никаких преподавателей — только задания, коллеги и открытые источники.

**Как работать с проектом:**

- Клонируй репозиторий с GitLab.
- Создай ветку `develop` — вся твоя работа ведётся только в ней.
- Помещай все файлы с решениями в папку `src/` внутри ветки `develop`.
- На проверку принимаются только файлы, которые находятся в папке `src/` ветки `develop`.
- Загружай только те файлы, которые указаны в задании.

**Подготовь рабочее окружение:**

Перед выполнением заданий необходимо создать базу данных. Для этого запусти скрипт из папки `datasets/`:

```
python datasets/create_db.py
```

Скрипт создаст файл `clinic.db` в той же папке. Этот файл — твоя рабочая база данных для всего проекта. Файл `clinic.db` не нужно коммитить в репозиторий.

Для работы с SQLite рекомендуем один из бесплатных инструментов:
- **DBeaver** (универсальный клиент, подходит для большинства СУБД)
- **DB Browser for SQLite** (легковесный, заточен специально под SQLite)

**Используй все доступные ресурсы:**

Гугли, читай документацию SQLite, задавай вопросы ИИ-ассистентам — это не запрещено, это часть процесса. Но будь критичен к источникам: проверяй, анализируй, сравнивай. То, что выдаёт ИИ или первая ссылка в поиске, не всегда верно.

**Проси помощи — и сам помогай:**

Если застрял — обращайся к коллегам. Если видишь, что кто-то застрял, предложи помощь. Обмен идеями — это не читинг, это суть peer-to-peer обучения.

**Не копируй — понимай:**

Скопировать готовый запрос легко. Но на защите тебя спросят: почему ты написал именно так? Что вернёт этот запрос? Что изменится, если убрать `WHERE`? Понимание важнее результата. Если ты понял, как работает JOIN, — ты можешь объяснить это другому. Если просто скопировал — нет.

**Формат сдачи:**

Каждое задание — один файл `.sql` с именем `task1.sql`, `task2.sql`, ..., `task6.sql`. Каждый запрос в файле предваряется комментарием с номером вопроса:

```sql
-- Вопрос 1:
SELECT ...

-- Вопрос 2:
SELECT ...
```

Все 6 файлов кладутся в папку `src/` ветки `develop`.

## Chapter II — Задания

### Схема базы данных

База данных `clinic.db` содержит шесть таблиц:

```
specialties   — медицинские специализации
branches      — филиалы клиники
patients      — пациенты
doctors       — врачи
services      — медицинские услуги
appointments  — записи на приём
```

Связи между таблицами:

```
appointments.patient_id  → patients.patient_id
appointments.doctor_id   → doctors.doctor_id
appointments.service_id  → services.service_id
doctors.specialty_id     → specialties.specialty_id
doctors.branch_id        → branches.branch_id
services.specialty_id    → specialties.specialty_id
```

Прежде чем приступать к заданиям, исследуй схему: посмотри на таблицы, изучи столбцы, загляни в данные (`SELECT * FROM ... LIMIT 10`).

### Задание 1: Первый допрос

**Для выполнения нужно знать:** `SELECT`, `FROM`, `WHERE`, `ORDER BY`, `DISTINCT`, `LIMIT`, операторы сравнения и `AND/OR/NOT`.

Руководство клиники открыло утренний брифинг. На столе — пять вопросов. Аналитик должен найти ответ на каждый из них, используя только одну таблицу за раз. Добро пожаловать в мир SQL: твой первый допрос данных.

**Напиши SQL-запросы, отвечающие на следующие вопросы:**

1. Выведи список всех пациентов из района **«Центральный»**, отсортированных по фамилии от А до Я. Покажи: `patient_id`, `first_name`, `last_name`, `district`.

2. Найди все записи на приём со статусом **«Отменён»** и выручкой (`revenue`) выше **3000 рублей**. Выведи: `appointment_id`, `appointment_date`, `revenue`.

3. Выведи **уникальный** список всех городов, в которых есть филиалы клиники. Столбец назови `город`.

4. Покажи **топ-10 приёмов с наибольшим временем ожидания** (столбец `wait_time_minutes`). Выведи `appointment_id` и `wait_time_minutes`, отсортируй по убыванию.

5. Найди всех врачей, нанятых **после 1 января 2020 года**. Выведи: `doctor_id`, `first_name`, `last_name`, `hire_date`. Отсортируй по дате найма от новых к старым.

**Результат:** файл `task1.sql` в папке `src/`. Каждый запрос предваряй комментарием `-- Вопрос N:`.

### Задание 2: Паспорт специализации

**Для выполнения нужно знать:** агрегатные функции (`COUNT`, `SUM`, `AVG`, `MAX`, `MIN`), `GROUP BY`, `HAVING`, псевдонимы (`AS`), `ORDER BY` с агрегатом.

Финансовый директор хочет понять, какие специализации тянут клинику вперёд, а какие отстают. Но смотреть на каждую запись по отдельности — не вариант. Нужны сводные показатели. Здесь вступают в игру агрегатные функции.

**Напиши SQL-запросы, отвечающие на следующие вопросы:**

1. Для каждой специализации (`specialty_id`) посчитай **количество проведённых приёмов** (статус `«Проведён»`) и **суммарную выручку**. Отсортируй по убыванию суммарной выручки. Выведи: `specialty_id`, `кол-во приёмов`, `суммарная выручка`.

2. Найди специализации, у которых **средний чек** (средняя выручка за один приём) превышает **4000 рублей**. Используй `HAVING`. Выведи `specialty_id` и среднюю выручку.

3. Для каждого врача посчитай: **количество уникальных пациентов**, которых он принял, и его **минимальную** и **максимальную** выручку за один приём. Оставь в результате только тех врачей, у которых было **не менее 10 приёмов** (любого статуса). Выведи: `doctor_id`, `уникальных пациентов`, `мин выручка`, `макс выручка`.

4. Найди **филиал с наибольшим средним временем ожидания** пациентов. Выведи `branch_id` и среднее время ожидания.

5. Посчитай, **сколько приёмов** приходится на каждый статус (`«Проведён»` / `«Отменён»`), и выведи их **долю в процентах** от общего числа приёмов. Результат округли до одного знака после запятой.

**Результат:** файл `task2.sql` в папке `src/`.

### Задание 3: Архитектура связей

**Для выполнения нужно знать:** `INNER JOIN`, `LEFT JOIN`, `FULL OUTER JOIN` (или эмуляция через `UNION`), соединение более двух таблиц, `ON`, внешние и первичные ключи.

До сих пор ты работал с одной таблицей за раз. Но реальные вопросы редко умещаются в одну таблицу. Имя пациента — в `patients`, имя врача — в `doctors`, специализация — в `specialties`. Чтобы собрать картину целиком, нужно научиться соединять таблицы.

> **Примечание:** SQLite поддерживает `FULL OUTER JOIN` начиная с версии 3.39 (2022 г.). Если твоя версия старше — используй эмуляцию через `UNION`. Проверить версию: `SELECT sqlite_version();`.

**Напиши SQL-запросы, отвечающие на следующие вопросы:**

1. Выведи список всех приёмов с полными данными: **имя и фамилия пациента**, **имя и фамилия врача**, **название специализации**, **дата приёма**, **статус**, **выручка**. Используй соединение четырёх таблиц.

2. Выведи всех врачей, **включая тех, у кого ещё не было ни одного приёма**. Покажи: `doctor_id`, `first_name`, `last_name` и количество приёмов (для тех, у кого их нет — `0`). Используй `LEFT JOIN`.

3. Выведи все **услуги** и количество раз, когда они были назначены на приёме. **Включи услуги, на которые ещё не было ни одной записи** (для них покажи `0`). Отсортируй по убыванию количества назначений.

4. Выведи всех **пациентов без приёмов** и все **приёмы без пациента** (если таковые существуют) в одном результирующем наборе. Используй `FULL OUTER JOIN` или эмуляцию через `UNION`. Добавь столбец `тип`, который указывает: `«пациент без приёма»` или `«приём без пациента»`.

5. Для каждого **филиала** выведи: название филиала (`branch_name`), количество уникальных врачей, работающих в нём, и суммарную выручку всех приёмов у врачей этого филиала. Учти, что у некоторых врачей может не быть приёмов — суммарная выручка для них должна быть `0`.

**Результат:** файл `task3.sql` в папке `src/`.

### Задание 4: Вложенный следователь

**Для выполнения нужно знать:** подзапросы в `WHERE`, подзапросы в `FROM` (производные таблицы), скалярные подзапросы в `SELECT`, `IN`, `EXISTS`, коррелированные подзапросы.

Иногда один запрос внутри другого — это самый точный инструмент. Как матрёшка: внешний запрос спрашивает, внутренний — отвечает. Главное — понять, какой подзапрос куда и зачем.

**Напиши SQL-запросы, отвечающие на следующие вопросы:**

1. Найди пациентов, у которых **суммарная выручка** по всем их приёмам **превышает среднюю** суммарную выручку по всем пациентам. Выведи `patient_id` и суммарную выручку пациента. Используй подзапрос в `WHERE`.

2. Для каждого врача выведи его **суммарную выручку** и его **долю (в %) от суммарной выручки его специализации**. Долю вычисли с помощью скалярного подзапроса в `SELECT`. Округли до одного знака.

3. Выведи **топ-3 услуги** по средней выручке приёма, на котором они назначались. Используй подзапрос в `FROM` как производную таблицу. Выведи `service_id`, `service_name`, среднюю выручку.

4. Найди пациентов, которые побывали хотя бы на **двух разных специализациях**. Используй подзапрос с `IN` или `EXISTS`. Выведи `patient_id`.

5. Найди врачей, у которых выручка за **последний месяц** в базе данных превышает их **собственное среднее** за всё время. Используй коррелированный подзапрос. Выведи `doctor_id`, выручку за последний месяц и среднюю выручку за всё время.

**Результат:** файл `task4.sql` в папке `src/`.

### Задание 5: Хроники клиники

**Для выполнения нужно знать:** функции работы с датами в SQLite (`strftime()`, `julianday()`, `date()`), `CASE WHEN ... THEN ... ELSE ... END`, вычисляемые столбцы.

Данные хранят время. Каждая запись на приём — это точка на временной шкале. А пациент — это история визитов. Умение работать с датами превращает сухой журнал в живую хронику клиники.

**Напиши SQL-запросы, отвечающие на следующие вопросы:**

1. Рассчитай **текущий возраст** каждого пациента в полных годах на дату выполнения запроса. Выведи `patient_id`, `first_name`, `last_name`, `birth_date` и возраст в годах. Отсортируй по убыванию возраста.

2. Разбей всех пациентов на **возрастные группы**: `«до 18»`, `«18–40»`, `«41–60»`, `«60+»` с помощью `CASE WHEN`. Для каждой группы выведи количество пациентов и суммарную выручку по их приёмам.

3. Выведи **количество приёмов и суммарную выручку по каждому месяцу** за весь период наблюдений. Формат месяца: `YYYY-MM`. Отсортируй хронологически.

4. Для каждого приёма добавь столбец `comfort_level` на основе времени ожидания:
   - `«критично»` — если ожидание превышает 60 минут
   - `«высокое»` — если от 45 до 60 минут включительно
   - `«норма»` — если менее 45 минут
   
   Выведи `appointment_id`, `wait_time_minutes` и `comfort_level`.

5. Определи, в какой **день недели** происходит наибольшее количество **отменённых** приёмов. Используй `strftime('%w', appointment_date)` (0 — воскресенье, 1 — понедельник, ..., 6 — суббота). Выведи день недели и количество отмен.

**Результат:** файл `task5.sql` в папке `src/`.

### Задание 6: Оконный ревизор

**Для выполнения нужно знать:** оконные функции (`OVER`, `PARTITION BY`, `ORDER BY` внутри окна), `ROW_NUMBER()`, `RANK()`, `DENSE_RANK()`, `LAG()`, `LEAD()`, накопленная сумма (`SUM() OVER`).

До сих пор агрегация сворачивала строки в одну. Оконные функции — другое: они вычисляют значение для каждой строки, не теряя детали. Каждый врач видит себя в контексте коллег. Каждый визит — в контексте предыдущего. Это и есть оконный анализ.

**Напиши SQL-запросы, отвечающие на следующие вопросы:**

1. Пронумеруй все приёмы каждого пациента в **хронологическом порядке** с помощью `ROW_NUMBER()`. Используй `PARTITION BY patient_id`. Затем выведи только **первый визит** каждого пациента (те строки, у которых номер = 1). Покажи `patient_id`, `appointment_date`, `doctor_id`.

2. Составь **рейтинг врачей внутри каждой специализации** по суммарной выручке с помощью `RANK()`. Выведи только тех, кто занимает **1-е место** в своей специализации (включая одинаковые результаты — тай). Покажи `specialty_id`, `doctor_id`, суммарную выручку и ранг.

3. Для каждого приёма каждого врача выведи **выручку предыдущего приёма** того же врача (в хронологическом порядке) с помощью `LAG()`. Вычисли **разницу** между текущей выручкой и предыдущей. Если предыдущего приёма нет — покажи `NULL`. Выведи `doctor_id`, `appointment_date`, `revenue`, предыдущую выручку и разницу.

4. Рассчитай **накопленную (кумулятивную) выручку** каждого врача в хронологическом порядке его приёмов (`SUM() OVER` с рамкой `ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW`). Выведи `doctor_id`, `appointment_date`, `revenue`, накопленную выручку.

5. Используя `LEAD()`, определи **количество дней до следующего визита** того же пациента. Выведи только тех пациентов, у которых хотя бы один межвизитный интервал **превысил 90 дней**. Покажи `patient_id`, `appointment_date`, дату следующего визита и интервал в днях.

**Результат:** файл `task6.sql` в папке `src/`.
